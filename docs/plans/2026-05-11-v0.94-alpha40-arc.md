## Sprint 28 — v0.98.0/176: UnifiedPush + ntfy-compat SSE

**Closes:** #122  
**Version:** 0.97.0/175 → 0.98.0/176  
**Daemon ref:** alpha.35  
**Touch points:** `AppRoot.kt`, `TransportClient.kt`, `RestTransport.kt`, `ServerProfileStore.kt`  
**Architecture decision:** UnifiedPush SSE supplements ntfy (both run in parallel). Base alpha.35 only (`waiting_input`). Alpha.35a (council/error/algorithm events) deferred pending parent-project topic-taxonomy confirmation.

### Scope

#### App-start registration
On app start (per enabled server profile):
1. Generate stable `client_id` = `UUID` stored in `EncryptedSharedPreferences` keyed by server URL
2. POST `/api/push/register` with `{endpoint: "<app-local-sse-url>", client_id: "<uuid>", token: "<bearer>"}`
3. Registration is fire-and-forget; failure logged but does not block startup

#### Topic subscription
Subscribe to the `alerts` catch-all topic via SSE:
`GET /api/push/alerts` (persistent connection, ntfy-compat SSE format)

SSE event format:
```
event: message
data: {"title":"…","message":"…","priority":4,"tags":["session-id"],"click":"/sessions/abc"}
```

Handle `event: open` (connection ack) and keepalive (25s ping — no action needed).
Reconnect with exponential backoff (1s → 2s → 4s → 30s max) on disconnect.

#### Notification rendering
On `message` event:
- Priority 4–5 → `NotificationCompat.PRIORITY_HIGH` (heads-up notification)
- Priority 1–3 → `NotificationCompat.PRIORITY_DEFAULT`
- `click` field → deep-link `PendingIntent` into the app's session detail route
- `tags` array → used to label the notification channel / group by session

#### Relationship to ntfy
ntfy wake-on-alert path (`WearSyncService` + Wear `WearAlertListenerService`) remains unchanged.
UnifiedPush SSE adds a parallel rich-notification path with deep-link support.
Both fire independently — this is intentional redundancy, not double-delivery.
