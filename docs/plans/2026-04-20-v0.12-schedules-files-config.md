## Phases ÔÇö implementation order

Transport first, then per-surface UI, then docs + release. Each UI
surface lands in its own commit so rollback granularity matches v0.11.

### Phase 1 ÔÇö `:shared` transport additions

Six new `TransportClient` methods. Additions-only (no interface
breakage). Tests mirror the v0.11 MockWebServer pattern.

1. `listSchedules(): Result<List<Schedule>>` Ôćĺ `GET /api/schedule`
2. `createSchedule(task: String, cron: String, enabled: Boolean = true): Result<Schedule>`
   Ôćĺ `POST /api/schedule`
3. `deleteSchedule(id: String): Result<Unit>` Ôćĺ `DELETE /api/schedule?id=<id>`
4. `browseFiles(path: String? = null): Result<FileList>` Ôćĺ `GET /api/files?path=<path>`
5. `listCommands(): Result<List<SavedCommand>>` Ôćĺ `GET /api/commands`
6. `saveCommand(name: String, command: String): Result<Unit>` Ôćĺ `POST /api/commands`
7. `deleteCommand(name: String): Result<Unit>` Ôćĺ `DELETE /api/commands?name=<name>`
8. `fetchConfig(): Result<ConfigView>` Ôćĺ `GET /api/config` (masked fields stay as
   sent; mobile renders `***` verbatim)

New domain types:
- `Schedule(id, task, cron, enabled, createdAt)`
- `FileEntry(name, path, isDirectory)` + `FileList(path, entries)`
- `SavedCommand(name, command)`
- `ConfigView(raw: Map<String, JsonElement>)` ÔÇö permissive wrapper since the
  parent's config schema is large and version-skewable

**Tests:** MockWebServer round-trip per method (~10 new cases).

**Acceptance:** `:shared:testDebugUnitTest` green with 10 new test
functions; no UI code touched.

### Phase 2 ÔÇö Schedules surface

Settings Ôćĺ new **Schedules** card listing every scheduled command with
per-row delete, plus a **+ New schedule** affordance opening a dialog
with:

- Task text (multi-line)
- Cron expression (with inline hint ÔÇö "`0 9 * * *`" etc.)
- Enabled toggle

Session detail gets a "Schedule reply" entry in its overflow menu so
a user can turn a reply into a recurring nudge.

**Why Settings and not a new tab:** the bottom nav is already four tabs;
adding a fifth for a rarely-used feature burns density. Put it with the
other server-state cards in Settings.

### Phase 3 ÔÇö File picker

Reusable `FilePickerDialog` composable:

- Top: current path breadcrumb, tap a crumb to jump up
- List: entries sorted dirs-first; `ÔČć ..` parent row; tap a dir to
  descend; tap a file to select
- Bottom: **Pick this folder** button when no file is selected

Wires into two call sites:
1. `NewSessionScreen` ÔÇö adds a "Working directory" field with a
   "BrowseÔÇŽ" button (currently new sessions inherit the server's cwd).
2. `ScheduleDialog` (Phase 2) ÔÇö same Browse button for the schedule's
   working dir.

### Phase 4 ÔÇö Saved command library (BL20 absorbed)

Settings Ôćĺ new **Saved commands** card. Tap a row to expand; swipe to
delete. `+` opens a "Save command" dialog with name + command fields.

Integration with `NewSessionScreen`: a "From library" dropdown above
the task field inlines the saved command's body into the editor so the
user can tweak before starting.

Closes BL20. Moves to `Completed backlog`.

### Phase 5 ÔÇö Config viewer (read-only)

Settings Ôćĺ new **Daemon config** card with a collapsible
key-value tree. Masked values render as `***` exactly as the parent
sends them. A subtle "Read-only ÔÇö write lands in v0.13 via a
structured form per ADR-0019" footer.

### Phase 6 ÔÇö Session output backlog pager UI

In `SessionDetailScreen`'s terminal toolbar (v0.11 Phase 3), add a
**Load backlog** icon to the left of the search icon. On tap: call
`transport.fetchOutput(sessionId, lines = 1000)` and prepend the text
into the xterm buffer via a new `window.dwPrependBacklog(s)` host
function. Idempotent ÔÇö pressing it again is a no-op until the session
changes.

**Why prepend and not clear-then-replay:** the WebSocket stream is
already writing live events; replacing the buffer would strand any
frames that arrived during the pager call. Prepend = "here is what
came before you subscribed".

## Risks

| Risk | Impact | Mitigation |
|---|---|---|
| Parent `POST /api/schedule` wire shape differs from audit | Phase 2 transport mismatch | Live-call the parent with a curl before shipping the DTO; fall back to permissive `JsonElement` body if fields don't match |
| `/api/files` treats some paths as restricted | Phase 3 picker shows "access denied" | Surface the server's 403 as an inline row message, don't crash |
| Config `***` masking omits some sensitive fields that the parent hasn't flagged yet | viewer shows a real token | Apply a second client-side mask to anything matching common secret-field-name patterns (`*token*`, `*secret*`, `*key*`) before display |
| Version skew: `/api/commands` CRUD doesn't match BL20 user-intent model | Phase 4 awkwardness | Start with list + save + delete only; don't try to model categories/tags until BL21 phase 2 |
