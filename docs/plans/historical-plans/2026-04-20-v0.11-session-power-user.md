## Goal

Bring session-detail and supporting surfaces to the level a power user expects
on the PWA: rename / restart / delete sessions, connection-status awareness,
daemon version visibility, CA-cert download for self-signed TLS, terminal
search + copy, explicit alerts dismissal, start-session form, active-backend
picker. Closes the 8 items in `parity-plan.md` §6 v0.11 plus three
opportunistic pickups the re-audit surfaced.

## Phases — implementation order

Transport first (narrowest blast radius), then UI, then xterm asset swap,
then live-validation. Each phase ships a separate commit and can be
individually reverted without breaking others.

### Phase 1 — `:shared` transport additions (no UI yet)

New `TransportClient` methods + `RestTransport` impls. Keep the interface's
"load-bearing" rule in mind — these are **additions**, not changes, so they
are safe per AGENT.md versioning:

1. `renameSession(id: String, name: String): Result<Unit>` → `POST /api/sessions/rename`
2. `restartSession(id: String): Result<Unit>` → `POST /api/sessions/restart`
3. `deleteSession(id: String): Result<Unit>` → `POST /api/sessions/delete`
4. `deleteSessions(ids: List<String>): Result<Unit>` — bulk variant; maps to
   parallel single deletes if the parent endpoint doesn't support list bodies
   (verify via live server call during Phase 1 test).
5. `fetchCert(): Result<ByteArray>` → `GET /api/cert` (returns PEM bytes)
6. `setActiveBackend(name: String): Result<Unit>` → `POST /api/backends/active`
   — **parent-confirmation gate:** check live server response shape before
   shipping; if endpoint doesn't exist, open upstream issue and skip this
   item until parent lands it.
7. `listAlerts(): Result<List<Alert>>` → `GET /api/alerts` (new `Alert`
   domain type in `:shared`)
8. `markAlertRead(id: String): Result<Unit>` → `POST /api/alerts`
9. `fetchInfo(): Result<ServerInfo>` → `GET /api/info` (hostname, version,
   session count — surfaces daemon version + hostname cleanly for About card)
10. `fetchOutput(sessionId: String, lines: Int = 500): Result<String>` →
    `GET /api/output` — opportunistic pickup; useful as a backlog pager for
    sessions that predate the WebSocket subscription. Defer wiring to
    Phase 3 UI.

**Tests:** MockWebServer round-trip test per method, including one
`TransportError` failure mode (Unauthorized / ServerError). Pattern
matches existing `RestTransportTest`.

**Not in this phase:** `ServerInfo.version` → About card wiring (Phase 2
UI), cert → OS trust store import flow (Phase 2).

**Acceptance:** `./gradlew :shared:testDebugUnitTest` passes with ~20 new
test cases; `TransportClient` has 10 new method signatures; zero UI code
touched.

### Phase 2 — Android UI wiring (composeApp)

Every new surface re-uses the existing Material 3 patterns. No new nav
destinations — everything slots into existing screens.

2.1 **Session row overflow menu** (`SessionsScreen.kt`). Three-dot on each
row → Rename / Restart / Delete. Rename = dialog with text field + cancel/save.
Restart = confirm dialog (server does a warm-reuse so no data loss, but confirm
the user intent — it can be disruptive if the PTY is mid-stream). Delete =
confirm with destructive wording; soft-disable if `state == Running`.

2.2 **Bulk actions** (`SessionsScreen.kt`). Long-press enters multi-select
mode; TopAppBar shows count + Delete action + Cancel. Limited to Delete for
v0.11 — bulk rename/restart is out of scope.

2.3 **Connection status indicator** (`SessionsScreen.kt` + shared TopAppBar
util). Observe `TransportClient.isReachable` — render an 8 dp dot next to the
title: green (ok), grey (probing), red (unreachable > 30 s per ADR-0013). Tap
the dot → bottom sheet showing last probe timestamp + retry button.

2.4 **About card — daemon version + hostname row** (`SettingsScreen.kt`).
Query `fetchInfo()` on screen enter; render "Connected to <hostname> · datawatch
vX.Y.Z" below the existing app-version row. Fallback to em-dash if unreachable.

2.5 **CA cert download helper** (`SettingsScreen.kt` servers card). Each server
row gets an overflow menu with "Download CA cert". Tap → `fetchCert()` → saves
to `MediaStore.Downloads/datawatch-<profile>-ca.pem` + opens Android's system
"Install CA cert" screen via `Settings.ACTION_SECURITY_SETTINGS` intent. We
can't silently trust-anchor the cert on unrooted Android — this hands the user
to the OS flow that does. Document the extra step in `docs/installation.md`.

2.6 **Start-session form** (new `NewSessionScreen.kt` + nav route). FAB on
Sessions tab → form: task text (multiline), optional server picker (defaults
to active), optional backend picker (reuses existing `listBackends()`). Tap
Start → `transport.startSession(task, serverHint)` → navigate to detail.
Voice reuse: the composer mic button from SessionDetail is lifted into a
shared `VoiceInputRow` so New Session gets it too.

2.7 **Active backend picker** (`SettingsScreen.kt` or Channels tab —
deciding at implementation time based on layout density). Radio-list of
available backends, `setActiveBackend()` on tap. Disabled with tooltip if
the parent-confirmation gate (Phase 1 item 6) failed.

2.8 **Explicit alerts dismiss** (`AlertsScreen.kt`). Swipe-left on each
row → red Dismiss action → `markAlertRead(id)`. Existing swipe-to-mute
stays; the gestures are on opposite sides (mute left, dismiss right) to
match Gmail/Discord conventions.

**Tests:** deferred. `composeApp` has no androidTest sourceset today —
adding the Compose UI test infrastructure (dependencies, runners, fake
`TransportClient` scaffolding) is a separate task. Phase 4
live-validation is the acceptance gate for Phase 2; UI-test setup is a
follow-up commit after v0.11 ships.

**Acceptance:** all 8 UI flows reachable from a clean install; accessibility
pass (TalkBack labels, 48 dp touch targets) verified in Phase 4 live-test.

### Phase 3 — xterm search + copy (xterm WebView asset swap)

1. Vendor `xterm-addon-search@0.15.0` into `composeApp/src/androidMain/assets/xterm/`
   (matches xterm@5.3.0 ABI per the xterm compatibility matrix — verify against
   the npm tarball's `peerDependencies` before committing).
2. `host.html`: import + `term.loadAddon(new SearchAddon())`; expose
   `window.dwSearch(query, opts)` / `window.dwSearchNext()` / `window.dwSearchClear()`
   to the Kotlin bridge.
3. Terminal toolbar above xterm (new composable): search-icon button → inline
   search bar with prev/next arrows + close. Clipboard button: copies the
   current terminal selection via `window.dwCopySelection()` → Kotlin-side
   `ClipboardManager.setPrimaryClip`.
4. Verify the search overlay doesn't interfere with the B1 write-cursor fix
   (`sessionId`-keyed `lastWrittenIndex` in `TerminalView.kt`) — the addon
   works on the scrollback buffer, not on the write path.

**Acceptance:** Search in a session with >200 lines of backlog highlights
matches in under 100 ms; copy-to-clipboard round-trips through `adb shell
cmd clipboard dump`.

## Opportunistic pickups from the 2026-04-20 PWA re-audit

PWA exposes these that aren't on the v0.11 parity-plan line. In-scope:

- `GET /api/info` — powers Phase 2.4 About-card hostname+version row cleanly
  (better than inferring from `/api/health`). Included in Phase 1 item 9.
- `GET /api/output` — backlog pager. Transport stub ships in Phase 1 (item 10);
  UI deferred to v0.12 alongside `/api/sessions/timeline`.

Flagged for user decision (not in v0.11 by default):

- `GET/POST/DELETE /api/commands` — saved-command library (user can save
  `new: refactor X to Y` and recall later). Good mobile QoL feature but needs
  UX design. **Recommend: BL20, target v0.12 with file picker + session
  prefs.**
- `POST /api/link/start` + `GET /api/link/stream` + `GET /api/link/status` —
  Signal device linking via QR code. Lives in the "messaging backends"
  setup story, belongs with Channels tab work. **Recommend: BL21, target
  v0.12 channels work.**

## Wear + Auto impact

v0.11 is phone-focused. Wear surfaces do not get rename/restart/delete
(small screen + driving-safety argument). Connection-status indicator
**is** mirrored into the Wear dashboard title bar (trivial — same
`isReachable` flow). Android Auto stays Messaging-template only.

## Risks

| Risk | Impact | Mitigation |
|---|---|---|
| `/api/backends/active` not yet in parent | Phase 1 item 6 blocked | Open upstream issue, gate 2.7 UI behind the transport availability check — greyed-out with tooltip pointing at the parent issue |
| xterm-addon-search ABI mismatch with vendored xterm@5.3.0 | Phase 3 blocked | Match versions per peerDependencies before commit; if incompatible, bump xterm to the addon's required minor and live-validate existing terminal flows as a sub-phase |
| Destructive delete used accidentally | user data loss | Two-step confirm with destructive-styled button; no swipe-to-delete in v0.11 (menu only); bulk delete requires long-press → multi-select mode |
| CA cert handoff to OS trust store fails on some OEMs | Phase 2.5 partial | Ship the download + MediaStore placement even if intent handoff fails; user can install manually from the system Settings app |

## Versioning

  to `0.11.0` in the final release commit of Phase 4.
- `versionCode`: 11 → 12 (monotonic per AGENT.md).
- Tag `v0.11.0` after Phase 4 sign-off. Release workflow attaches AABs +
  APKs + SHA256SUMS.
